can an APPROVED w/o be "un-APPROVED" ?  

Before a w/o is transmitted, it can be changed from APPROVED to SUSPENDED by adding "internal remarks" and clicking the SUSPEND button at the top of the form.

 Any w/o that shows status APPROVED can be changed to either SUSPENDED or REJECTED. This is done by clicking the appropriate button at the top of the w/o screen. MERC will prompt you to add the appropriate remarks if you have not done so. Keep in mind a suspended w/o can be APPROVED or REJECTED. But once a w/o is REJECTED - it must be resubmitted by the vendor.

Prep Time - where/how does it apply ?

  • Prep time (preparation time) is allowed on some modes (usually those beginning with 0) and for most, but not all, repairs within those modes.
    • Exception to rule is for cleaning, nail or label removal etc. If this is all that is done unit - preptime is not allowed.
  • It is calculated automatically and given to the vendor
  • RULES FOR CALCULATING PREP TIME
    • if all codes are eligible for p/t then prep time is calculated on total hours.
    • if no codes are eligible for p/t then prep time is not calculated.
    • if at least one code is eligible for p/t then the p/t is calculated on total hours of estimate (eligible and non-eligible codes.
  • INTERNET PREP TIME - MERC+ will calculate the prep time and insert the valid amount on the estimate/work order. This prep time will be calculated at the lowest rate on the estimate/work order (i.e. straight time if present).
  • EMAIL and WEB SERVICES PREPARATION TIME
    • If Email or Web Services transmission has an estimate which is eligible for prep time and a prep time code (00xx) is present - MERC+ will validate that prep time code and accept or reject according to the rules.
    • If Email or Web Services transmission has an estimate which is eligible for prep time and a prep time code (00xx) is NOT present - MERC+ will calculate the prep time code at the lowest rate on the estimate/work order (i.e. straight time if present).

Shortcuts in w/o entry

MULTIPLE BROWSER SESSIONS

  • You can log into MERC on multiple browser sessions simultaneously. This means if you hit a problem, such as bad unit number or question that can't be answered immediately, you may start a new session and continue inputting other work order's without losing work you have done on the w/o in question. If however you cannot complete the input of the work order by the end of the day - you will lose the work that has not been transmitted.
  • You can switch from session to session if you have multiple browsers open - or if you have other applications running (email, spreadsheets etc.). This can be done in 3 ways:
    • Hold down the ALT key and press TAB key
    • Hold down the ALT key and press ESC key
    • Select session from bottom of screen with mouse cursor.

RAPID ENTRY TO LIKE WORK ORDER'S

Work order's that are basically similar (such as FHWA's and Reefer pre-trips) can be entered with a minimum of keystrokes.

  1. First confirm you have multiple work order's that are the same except for unit number.
  2. Enter the first work order.
  3. Transmit the first work order.
  4. Hit the BACK key on your Browser.
  5. Type in new VENDOR REF#.
  6. Type in new UNIT NUMBER.
  7. -if- completion date is different - enter the new date.
  8. Transmit the 2nd work order.
  9. Go to step 4 and repeat...

Vendor Reference numbers - who creates/tracks them ?

Vendor reference numbers serve as both a work order number and an invoice number.

Vendor Ref # field in MERC is a "free" field - it can be any combination of letters and/or numbers up to 10. If more than 10 - only the 10 places of number will be returned on the checkstub.

Vendor will input and track vendor reference numbers. It is their responsibility to  create and use a numbering system which will allow them to track their work order's and payments.

DUPLICATE VENDOR REF NUMBERS are not allowed.  If you try to enter a w/o with a number which has already been used - MERC will not accept the transmission and will advise you that the reference number is already in the system.

The exception to this rule is when original w/o was rejected. If original w/o has been rejected - w/o number may be reused if desired with corrected w/o or with any other w/o. (Check with your accounting dept - but judge most will want to know about any "missing" w/o numbers in a series.)

MERC looks for unique numbers for a given shop. It is not necessary for numbers to be unique over shops (ie 2 or more shops can have a w/o numbered 22)

Overtime - how is an estimate entered with overtime ?

W/O's are created in straight time as a default. To change w/o to overtime (time and a half, double time etc.)

  1. Enter codes, shop materials and hours
  2. Click REVIEW button
  3. At bottom of new screen are total hours entered as straight time and blocks for other rates.
  4. Total of all blocks must equal the total of the w/o.

How are rate increases handled ?

There is provision in MERC for current rates & future rates. For example you can have a set of rates which expire 10/1/2001 and weeks or months before that date you can input 2nd set of rates which would become effective 10/2/2001. Rate chosen is governed by date of w/o (which is day work was completed). Date of input does not come into play.

In many cases rates won't be settled weeks in advance - often down to last minute. We can extend present rate day by day it needed and back date rates for work order's that have not been input. If a w/o has been input - rate cannot be changed. However, we must have information from field in order to keep rates current.

Rates are good until midnight of day of expiration. W/O's dated (input date is irrelevant) the day after expiration not be allowed - the error message will read 

Please Correct the Following Error(s) and Resubmit:
Labor Rates are not valid for the repair date indicated

    • Guidelines for update of off hire of leased equipment in MERC+

      With implementation of MERC+, new guidelines will apply for update of off hired leased equipment.
      Old guidelines used in RKRP
      Off hire cost to be inserted as a lumpsum under correct off hire mode (8X) using repair shop 998 and repair code 0080.

      Some repair shops (MSL shops) are repairing equipment prior redelivery (pre off hire repairs). These shops shall update the repairs under their own shop using STS, i.e. code 0950 is NOT to be used.

Why is right clicking not possible in MERC+, how do I then copy?

Right click is disabled in MERC+ for the following reason:
When users 'right click' and open a separate window, the nature of a web based application causes the web server not to view the new window as a new session, therefore, the backend component containing estimate information will not be updated. This in turn can compromise the entire system.

Copy paste is still possible though;
In order to cope use "CTRL+C"
In order to paste use "CTRL+V"

Why will MERC+ not accept the part number I am entering?

The part number table in MERC+ contains the full price lists from Carrier, TK and Daikin (approximately 33000 part numbers).

Specifically with regards to Carrier there is and has always been confusion with regards to how the part number is written. Carrier is often adding multiple dashes to their part numbers and may show a part number in a reefer unit part list as 69NT42-688-7 while the number shown on the box containing the spare part is 69NT--42---688-7.

Carrier has informed that 69NT--42---688-7 is the correct number and that this is the number that can be looked up in their global price list. There is no guarantee it is possible to look up 69NT42-688-7

Since MERC+ is using the global price list you should always enter the part number shown on the box as there is no guarantee the number from the spare part manual is in the price list.

You may try to use the part number from the spare part manual and if MERC+ accepts it everything is in order however there is just no guarantee.

3rd Party recovery update   CEN65049

For update of 3rd party recovery, RKRP03 is still to be used and report number 999999 shall be used for all MERC+ work orders. It is not possible to update recovery in MERC+ yet.

If no 3rd party is updated on work order when submitted from MERC+, then RKRP29 error log will have to be changed from M to Y or V by a RKRP super user in order to update the work order in RKRP01 with 3rd party port. This will have to take place before RKRP03 can be updated. When 3rd party port updated in RKRP the error log must be switched back to M again.

Note that this is only needed when the work order is submitted from MERC+ without 3rd party updated. If MERC+ is updated with 3rd then RKRP03 can be updated directly.

Note that switching back and forth between M and Y or V should be minimized. If MERC+ transmit work orders to RKRP while switch is set to Y or V, these will end up in RKRP70 error log. Therefore always check RKRP70 for work orders after you have changed error log switch. FYI; MERC+ transmits to RKRP every 15 minutes around the clock and you can never know whether it is just about to transmit or transmission has just taken place.

Expiration of Labor Rates and/or Material Contracts

When shops/countries material contracts and labour rates expire - unless a new contract valid from the the day after expiration date is uploaded before the expiration date, the following will happen:

1) Material contract expiring.
System will default to global ceiling.
Since system will no longer see a country contract CPH will pay shop material rates also for decentralized countries

2) Labour rates expiring
Shop will not be able to enter estimates at all for equipment where rates has expired. If only country rates has expired, CPH will pay shop labour rates to agent.

It is therefore of paramount importance you ensure all shop/country material and labour rates are ready BEFORE the existing rates expire.

Work orders in RKRP70
You have likely noticed an increasing number of work orders in RKRP70 screen for your shops. If the shop is using MERC+ they will all have work order number 999999 and if you open the work order and tries to insert it in RKRP message "IDENT. ALREADY EXISTS, SECURITY ERROR" meaning that the work order already exists in RKRP and cannot be uploaded.
The reason for work orders appearing as duplicates is as follow;
After a work order is completed in MERC+ it is transmitted to RKRP. Work orders are transmitted in batches every 15 minutes. MERC+ does not receive any feedback from RKRP and is thus unable to know if the work order was received in RKRP. If RKRP is down because of problems or maintenance, work orders from MERC+ would be lost unless precaution taken. In order to ensure no work orders are lost MERC+ will receive message from MQ series (interface between MERC+ and RKRP) that RKRP is unavailable. If MERC receives such message in the middle of a transmission to RKRP, transmission will stop immediately. When MQ series is reported up again, MERC+ will send the batch again. Some work orders inevitably will have made to RKRP before MQ went down. When MERC+ sends the same batch again, RKRP will thus receive some work orders that already were transmitted the first time and it is those duplicates you will find in RKRP70. they can just be deleted.

RKRP prime mover is working with IBM to have such duplicate work orders automatically deleted from RKRP70
