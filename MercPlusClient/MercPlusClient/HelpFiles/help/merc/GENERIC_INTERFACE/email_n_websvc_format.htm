EMAIL FORMAT FOR GENERIC INTERFACE

 

  • Explanation of the email format
     
  • Tester for email format (.xls file)
     
  • Format for email transmissions compared to AutoRKRP transmissions
     
  • w/o’s can be put into text file format. How often - transmissions can be received at any time. It is suggested that they take place nightly. Longer intervals will make error correction more difficult.
     
  • To send emails to MERC+, the vendor must be registered with enterprise.

Purpose
The purpose of this interface is to allow outside vendors to transmit Estimate information to MERC+ via a file attached to an email.  The email will be received in Enterprise, validated for format, and if successful, each Estimate within the file will be individually routed to MERC+ system  using EAP( Enterprise Application Protocol).

Requirements
The communication between the outside vendor and the Enterprise receiving process will be by any standard email facility which allows file attachments to email transmissions. 
The attached file must follow the format as defined below.  This file may be system generated or manually created as long as it conforms to the defined format.
Estimates may be sent individually on a file, or batched together under one header control record.
Each line of the message will be size delimited and must have 80 characters padded with spaces.
Each line must be filled capacity as follows:

  • Numbers      - with leading zerors,  100.00 in 12 digit field = <000000010000>
  • Labels         - with leading spaces,  A123  in 6 char field =  <  A123>

Functional Processing

The detailed steps that define how this data will be processed.

Function Processing Logic
  * * * * *  Process design as in existing MERC * * * * *

Polling process constantly monitors Enterprise mailbox for any new Estimates received

New Estimate received follows same processing as new online MERC+ logic

  • Equipment Number validated in RKEM, data retrieved
  • All validations as online process
  • Calculations as online process
Compare Totals received Final validation of online Estimate is to verify HD2.Total Amount against system calculated total.  However, email attachment will not include any taxes, nor will it include parts prices, so the total can only be compared against the total materials and total labor in MERC+ for Shop costs, not CPH costs. 

The other issue is that shop amount can now come in any currency, not just USD.  The allowable deviated amount between the amount transmitted in the WO and the system calculated amount is 1.00 USD.

For Processing

  • Get HD1.ShopCode.Currency_Cd.  If not USD, convert $1.00 into Shop local currency and save deviate amount.  If USD, save 1.00 as deviate amount
  • Compare HD2.TotalAmount  with ( WO.TOT_LABOR_COST  + WO.TOT_SHOP_AMT)  If difference is more than the deviated amount,  return error message # 20090

Interface Specification: MERC+ Web Service

 Last updated:  12/02/04

Purpose

MERC+ will provide a web service to create work orders for clients. The web service will accept a single work order for validation and/or insert and provide an accept reply or reject reply with stated reasons for failure.

Note: changes to this document are reflected by text that has been stricken through.  See the Change Log at the bottom.

requirements

Work orders sent to MERC+ via the web service must adhere to a pre-defined XML format or DTD (Document Type Definition) as provided in this document. The response XML will also be defined. Clients wishing to create work orders with MERC+ via the web service are responsible for developing and maintaining client based software to use the web service. It is expected that the client will use a digital certificate just as is required for the MERC+ web application.

Description  
Interface Type

 Asynchronous   /  web service
 Persistency

Request/response messaging. Only successfully processed work orders will persist in database.
 Expiry

Work orders which have failed validation will not be persisted. Results are returned immediately to the request process for handling. Expiration periods do not apply.

Delimiter format

Fields are defined in XML tags - no padding is required.
Individual Messages

Single work orders will be processed per request only
Performance Considerations

Performance is expected to be similar as is for the UI - same MERC+ components will be used to validate and persist a work order.
Design Considerations

Web service will be developed to run in the Microsoft .NET runtime
Message Design

Work order will be delivered in an XML format (DTD defined below)

Functional Processing

The detailed steps that define how this data will be processed.

Function Processing Logic
Send workorder to MERC+ via web service Web service will accept a request for a work order via HTTP. It will extract the certificate ID and validate against Get Access. Once the user is validated, the service will continue to parse the work order information from the XML document and populate the MercWorkOrder component with the work order data. It will call the components Validate method. If no errors are received from the validation, it will call the component's save method. Upon a successful save, the service will return a response to the client with an accepted status of 'YES'. If the estimate fails validation, the response will be sent with an accepted status of 'NO' and the error(s) will be included in the response document. Note: if the user cannot be validated against Get Access, the response will be returned with an accept status of 'NO' with a 'no authorization' error message.
