Interface Specification: MERC+ Web Service


Purpose

MERC+ will provide a web service to create work orders for clients. The web service will accept a single work order for validation and/or insert and provide an accept reply or reject reply with stated reasons for failure.

requirements

Work orders sent to MERC+ via the web service must adhere to a pre-defined XML format or DTD (Document Type Definition) as provided in this document. The response XML will also be defined. Clients wishing to create work orders with MERC+ via the web service are responsible for developing and maintaining client based software to use the web service. It is expected that the client will use a digital certificate just as is required for the MERC+ web application.

Description  
Interface Type

 Asynchronous   /  web service
 Persistency

Request/response messaging. Only successfully processed work orders will persist in database.
 Expiry

Work orders which have failed validation will not be persisted. Results are returned immediately to the request process for handling. Expiration periods do not apply.

Delimiter format

Fields are defined in XML tags - no padding is required.
Individual Messages

Single work orders will be processed per request only
Performance Considerations

Performance is expected to be similar as is for the UI - same MERC+ components will be used to validate and persist a work order.
Design Considerations

Web service will be developed to run in the Microsoft .NET runtime

Work order will be delivered in an XML format (DTD defined below)

Functional Processing

The detailed steps that define how this data will be processed.

Function Processing Logic
Send workorder to MERC+ via web service Web service will accept a request for a work order via HTTP. It will extract the certificate ID and validate against Get Access. Once the user is validated, the service will continue to parse the work order information from the XML document and populate the MercWorkOrder component with the work order data. It will call the components Validate method. If no errors are received from the validation, it will call the component's save method. Upon a successful save, the service will return a response to the client with an accepted status of 'YES'. If the estimate fails validation, the response will be sent with an accepted status of 'NO' and the error(s) will be included in the response document. Note: if the user cannot be validated against Get Access, the response will be returned with an accept status of 'NO' with a 'no authorization' error message.
