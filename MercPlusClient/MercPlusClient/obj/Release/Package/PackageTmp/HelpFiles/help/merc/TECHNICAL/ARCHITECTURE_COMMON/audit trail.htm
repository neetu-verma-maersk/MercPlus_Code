Merc+ Audit Trail Conceptual Design

NOTE - If the person hits the SUBMIT button without making any changes,  that person will show as updating on the screen. However, the audit trail will not be updated. The audit trail is only updated if data actually changes.

 

Merc+ will capture changes to  work orders and to some selected reference table fields.

Audit trail components will be developed specifically to manage audit trail record persistence. The business components will determine what information is to be captured. Database triggers will not be used.

 

Work Order Audit Trail

Audit trail capture for work orders will consist of all column changes including status events. User comments will not be included. Events will include all status changes such as:  created, approved, rejected etc. Audit trail records for a work order will be accessible in the user interface via an audit trail button.  The audit table will contain a CHANGECOMMENT field used primarily by the UI – this is a readable pre-formatted text description of the changes to be displayed to the user.

 

The following sample shows what a user will see when selecting the audit trail button for a work order:

 

  • Work order created:  WO number 00012345 created on 09/15/2003  08:23by John Smith
  • Part quantity changed from 2 to 1 for repair code 1234 on 09/17/2003   09:50 by John Smith
  • Work order status changed from suspended to approved on 09/20/2003   10:21 by Mary Doe

Reference Audit Trail

Audit Trail Components

Individual business component managers will be responsible for sending changes to the audit trail component. I.e., no database triggers will be used. The audit trail component will be designed to accept entries from one or more components and cache entries to be persisted to the database via a background thread. This will help minimize overall impact to the MERC+ system. 

 

Audit Trail and Archiving

Work order audit trail entries will remain with the associated work orders. I.e., when a work order is archived, its associated audit trail entries will be archived along with it.  It is expected that a batch process will be developed to perform archiving requirements. I.e., online transaction data must be available for a 3 year period and then  be moved to an archive database. We are assuming that this process could be run on a weekly basis.  The reference audit trail table will be archived after 3 months.
