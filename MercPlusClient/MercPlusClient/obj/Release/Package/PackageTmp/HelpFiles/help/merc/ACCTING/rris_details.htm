RRIS in original MERC

The current MERC process uses the following system table driven information to transmit invoices to RRIS.

  • Y/N switch to transmit to RRIS  - based on Customer Code and WO Mode
  • ACCOUNT_CD - based on Customer Code and WO Mode
  • PAYAGENT_CD  - based on Customer Code, Shop and Mode
  • RRIS screen Format to send (52, 70, 72)  - based on Payagent
  • Profit Center  - based on Payagent
  • Corporate Payagent Code  - based on Payagent
  • Sub Profit Center  - based on Payagent
  • Local Account Code - based on Payagent and Vendor
  • Supplier Code  - based on Payagent and Vendor
  • Accounting periods - based on Corporate Payagent Code.
  • Activity periods - based on Corporate Payagent Code

Depending upon the PayAgent Code set up in the system tables, one of the three following "screen" formats is populated via a file interface format. (The following definitions were taken from MERC document dated 5/3/2001)

RRIS52:

    Normally only used for payagent MSC and modes that begin with 0 or 2.  The payagent code will be assigned at the customer, shop mode level.  Only one Profit Center code is required.

RRIS70:

    Normally only used for payagent M99 and modes that begin with 1 or 3.  The payagent code will be assigned at the customer, shop, mode level.  A corporate Profit Center code is required for the payagent. A second Profit Center code (know as branch pc code) is required at the sub level.  This may or may not be the same as the corporate Profit Center code.

RRIS72:

    Can be used for any mode.   The payagent code will be assigned at the customer, shop, mode level and must be a branch payagent.  Branch payagent code must also have a corporate payagent assigned.  The corporate payagent (known as "payagent" ) will have a corporate pc code (known as pc code).   The branch payagent will have a sub level profit center code ( known as branch pc code) assigned, as in RRIS70.  This may or may not be the same as the corporate pc code. 

The RRIS72 format is used to combine work order payments from several branches(locations) of one company, so only one check needs to be sent per company.

An example is UMS.  They have locations in NWK, NFK, MIA and HOU. Each shop has it's own payagent code for it's customer, mode.  The UMS corporate code is MC1.  The branches are MC2, MC3, MC4, etc.  Usually only branches submit WO, but a branch can be a corporate too.  In that case, the RRIS70 format would be used.

Functional Processing

The detailed steps that define how this data will be processed.

Function Processing Logic
   Find work orders satisfying the following criteria:

if STATUS_CODE=400 (ACCEPTED) OR STATUS_CODE=550 (REJECTED)

 Generate a 100 start record

 For Each Work Order where the daily work order total is less than ten thousand:

  BEGIN

If the work order shop level RRIS_XMIT_SW='Y' and the transmit table RRIS_XMIT_SW='Y' then  increment the daily work order accumulator by 1

Determine the accounting period for the work order REPAIR_DTE field  If the current day within the current month is greater than   the ACTDAY_END value for the work order corporate pay agent then the accounting period month will be the repair date month plus one.  If adding a month to the repair date month changes the year then the account period year is the year following the repair date year.

Determine the activity period for the work order REPAIR_DTE field. The activity period year and month for a corporate pay agent is a function of the activity period start and end dates defined for that pay agent. If the repair date is greater than the activity period start date for one of the twelve defined activity periods, then use the activity start year and month associated with that period rather than the repair year and month.

  Generate a 101 header record

  Generate a 102 detail record

  Send to RRIS

  set the work order STATUS_CODE=500 (RRIS_SENT)

  END

  if the work order RKRP_XMIT_SW='N' then

  BEGIN

  set the work order STATUS_CODE=900 (COMPLETED)

  END

Security

Any rules or special constraints that deal specifically with security.

   
Rule Rule Description
File Format An RRIS upload file from MERC contains three sections in the following order:

One START record with MVS4 MERC UserID 'BNYCWMES' field

One or more HEADER records with '101' identifier and Description field containing left justified work order record pointer from DB

One or more DETAIL records with '102' identifier and a 52, 70 or 72 layout

HEADER/DETAIL records are always in pairs

Record length is 150 characters.  Default character value is blank unless specified otherwise in above layouts.

All text field data must be upper case
File Processing All of the required permissions and table look up values required by RRIS to respond to a payment request must exist at the time of receipt of the request.  If a table look up fails an error is sent to the internal RRIS error queue and no response is generated or voucher is rejected.  Need to have RRIS expert confirm what happens.

A maximum of ten thousand voucher requests can be submitted for a given calendar day (the day being defined by the time in Denmark).  Voucher numbers from MERC have the following layout :  'Z'+NNN+SSSS

NNN - number of days since the first day of the current year.  Three digits required.

SSSS -- Daily sequence number reset to zero each day.  Valid values '0000' thru '9999'

The concatenation of pay agent code plus voucher number plus account year
Daily Load MERC UserID BNYCWMES must be defined for all payagent codes for all the formats (52, 70, 72) that payagent will be used for.

RRIT49 and RRIT57 need to be updated for all payagents that MERC user BNYCWMES will be supporting in RRIS
